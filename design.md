# AstrBot 表情包偷取与识别插件设计

## 目标

监听群聊中的图片消息，在不阻塞 AstrBot 主消息流程的前提下，将图片交给视觉模型提取内容，再交给情景模型结合群聊文字选择 meme_manager 已有分类，最后保存到 meme_manager 的运行数据目录。

## 数据契约

插件通过 AstrBot 的数据路径 API 定位 `data/plugin_data/meme_manager`，不硬编码宿主机目录。图片保存为 `memes/<category>/<filename>`；分类描述文件 `memes_data.json` 只在缺少分类时补充默认描述，不覆盖用户已有内容。重复图片使用 SHA-256 判重。

## 识别流程

1. `on_message` 只处理群聊，并读取消息链中的图片组件。
2. 检查 `group_whitelist`：为空时允许所有群；非空时匹配群号或完整 UMO。
3. 下载/解码图片到插件临时目录，限制单图大小和每条消息图片数量。
4. 视觉模型返回图片描述；情景模型读取描述、群聊文本和可用分类，返回 JSON 分类。
5. 分类无效、模型异常或未配置时降级到 `confused`，仍保存图片并记录日志。

## 配置假设

`vision_provider_id` 和 `scene_provider_id` 为空时使用当前会话 Provider。默认开启自动偷取；默认允许全部群；默认不创建未知分类。插件不调用 meme_manager 的私有 Python 模块，避免其升级时发生导入耦合，只依赖其公开的文件目录契约。

## 依赖健康检查

插件将 meme_manager 视为软依赖：启动时检查 AstrBot 插件注册表，随后按间隔复查注册状态、`memes/` 目录读写权限和 `memes_data.json` 可解析性。健康状态不是 `ready` 时，图片事件直接返回但插件本身继续运行；`/表情偷取状态` 提供人工诊断入口。

## 统一发送出口

在 `on_decorating_result(priority=100000)` 中先清理 meme_manager 的 `&&category&&` 标记，阻止其发送处理器继续识别；随后读取机器人的纯文本回复，调用 `reply_scene_provider_id` 指定的情景模型返回 `should_send` 和 `category`。仅当 `should_send=true` 时进入对应分类，再将该分类的候选图片和索引交给多模态模型选择最符合语境的一张。未配置独立回复模型时复用 `scene_provider_id`。发送概率和会话冷却只由本插件配置控制。

## 已有库整理

表情库索引不再依赖手动命令。meme_manager 健康检查通过后，后台任务扫描每个现有分类目录；目录名是权威分类，多模态模型只记录图片描述、情绪、文字和标签，不返回目标分类，也不跨目录移动图片。未索引图片按批次提交，默认每批 6 张。每个目录拥有 `index.json` 和 `README.md`，图片只在原目录内按 `category_0001.ext` 编号。失败条目保留原文件并标记为待重试。

为减少 Gemini 按次调用，自动回复把 `should_send`、分类和候选图选择合并为一次多模态调用，并在调用前应用概率和冷却门控。

同一条群消息的多张图片使用一次批量视觉调用和一次批量情景分类调用；批量接口失败时才回退到逐张处理。
